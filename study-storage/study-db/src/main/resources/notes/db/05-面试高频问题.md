# 05-面试高频问题

## Q1: B+Tree 为何 3-4 层够存千万数据？

**答案要点：**

```
InnoDB 默认页大小 16KB:
  非叶子节点: 只存 key(8B) + 指针(6B) = 14B/条目
  每页约 1170 个索引条目

  叶子节点: 存完整行数据
  假设每行 1KB，每页约 16 行

层数计算:
  1层: 1 * 1170 * 16 = 18,720 行
  2层: 1170 * 1170 * 16 = 21,902,400 行 (约 2190 万)
  3层: 1170 * 1170 * 1170 * 16 ≈ 256 亿行

结论: 3 层可存约 2190 万行，千万级 3 层足够
      4 层可存约 256 亿行，覆盖绝大多数场景
```

**追问: 为什么不用二叉树/红黑树/哈希表？**

- **二叉树**: 高度 O(log2 N)，千万数据 24 层，意味着 24 次磁盘 IO
- **红黑树**: 自平衡但仍是二叉树，高度问题同二叉树
- **哈希表**: 不支持范围查询、排序、LIKE 前缀匹配
- **B+Tree**: 多叉 + 页对齐 + 叶子链表 = 3 次 IO + 支持范围查询

## Q2: Slow SQL 排查流程

```
1. 开启慢查询日志
   slow_query_log = ON
   long_query_time = 1 (秒)
   log_queries_not_using_indexes = ON

2. 定位慢 SQL
   - 慢查询日志文件分析
   - pt-query-digest 工具汇总
   - 监控平台 (Prometheus + MySQL Exporter)

3. EXPLAIN 分析
   EXPLAIN SELECT ...;
   重点看: type(ALL最差), rows(扫描行数), Extra(Using filesort/temporary)

4. EXPLAIN ANALYZE (MySQL 8.0+)
   实际执行并显示真实耗时

5. 优化方向:
   - 加索引: WHERE / JOIN / ORDER BY / GROUP BY 列
   - SQL 改写: OR->UNION ALL, 子查询->JOIN, 避免 SELECT *
   - 表结构: 垂直拆分, 避免大字段
   - 参数调优: join_buffer_size, sort_buffer_size
   - 架构: 读写分离, 分库分表, 缓存

6. 验证效果:
   EXPLAIN 对比优化前后
   压测确认 QPS 提升
```

### 慢 SQL 排查决策树

```
SQL 执行慢
  ├── 查看执行计划 type=ALL? -> 加索引
  ├── rows 很大? -> 加索引或改写 SQL
  ├── Extra=Using filesort? -> 优化 ORDER BY 索引
  ├── Extra=Using temporary? -> 优化 GROUP BY/DISTINCT
  ├── 走索引但 rows 仍大? -> 数据倾斜/索引选择问题
  └── 索引都OK? -> 表数据量大 -> 分表/归档/缓存
```

## Q3: 优化器走错索引怎么办？

**原因分析：**

1. 统计信息不准 (analyze table 未及时更新)
2. 数据倾斜导致索引选择性估算偏差
3. 优化器成本模型与实际不符
4. ORDER BY + LIMIT 中优化器误判

**解决方案：**

| 方案 | 语法 | 适用 |
|------|------|------|
| FORCE INDEX | `SELECT * FROM t FORCE INDEX(idx) WHERE ...` | 强制指定 |
| USE INDEX | `SELECT * FROM t USE INDEX(idx) WHERE ...` | 建议使用 |
| IGNORE INDEX | `SELECT * FROM t IGNORE INDEX(idx) WHERE ...` | 排除某索引 |
| ANALYZE TABLE | `ANALYZE TABLE t;` | 更新统计信息 |
| 优化器追踪 | `SET optimizer_trace='enabled=on';` | 查看为何选错 |
| 直方图 | `ANALYZE TABLE t UPDATE HISTOGRAM ON col;` | 解决数据倾斜 |

**面试话术**: "先确认是统计信息问题还是成本模型问题。优先用 ANALYZE TABLE 更新统计信息，80% 的情况能解决。不行再开 optimizer_trace 看具体为什么选错，最后才考虑 FORCE INDEX，但要记文档避免未来失效。"

## Q4: 分库分表怎么平滑扩容？

**关键点**: 避免全量数据迁移，减少停机时间。

**方案一: 倍数扩容 (推荐)**

```
2库 -> 4库 -> 8库:

步骤:
  1. 部署新库 (空库)
  2. 开启双写 (新旧两套集群都写)
  3. 历史数据按新分片规则迁移 (增量同步 + 全量迁移)
  4. 数据一致性校验
  5. 灰度切读流量 (逐步把读流量切到新集群)
  6. 全量切读写到新集群
  7. 下线旧集群

迁移量: 约 50% (user_id % 2N >= N 的迁移)
```

**方案二: 一致性哈希 (适合缓存)**

```
优点: 约 1/N 数据迁移
缺点: 热点数据问题，需要虚拟节点
适用: Redis, Memcached
不适用: 需要范围查询的 MySQL
```

**方案三: 停机迁移**

```
适用: 数据量 < 100GB，业务允许夜间停机窗口
流程: 停写 -> 导出 -> 导入 -> 切换配置 -> 恢复
优点: 简单，无一致性风险
缺点: 需要停机
```

## Q5: PostgreSQL vs MySQL 选型对比

| 维度 | PostgreSQL | MySQL |
|------|-----------|-------|
| 复杂查询 | 优秀(并行查询, CTE, 窗口函数) | 一般(8.0+ 有改进) |
| GIS | 内置 PostGIS | 有限支持 |
| 全文搜索 | 内置 GIN 索引 | 需要 Elasticsearch |
| JSON | JSONB 性能优秀 | JSON 功能较简单 |
| 复制 | 流复制/逻辑复制 | 主从/Galera |
| 扩展性 | 强大的扩展系统 | 插件较有限 |
| ORM 兼容 | Hibernate/JPA 支持好 | Hibernate/JPA 原生支持 |
| OLTP 简单读写 | 略慢于 MySQL | 更快 |
| 运维生态 | 相对少 | 成熟(PXC, MHA, Orchestrator) |

**选型建议:**

- **选 MySQL**: 简单 CRUD 多、读多写少、团队运维经验丰富、需要成熟高可用方案
- **选 PostgreSQL**: 复杂分析查询、GIS/时序/JSON、对数据一致性要求极高、需要扩展性
- **两者都选**: MySQL 做主业务 OLTP，PostgreSQL 做分析报表 OLAP

## Q6: Kingbase 迁移 MySQL 常见坑

| 坑点 | 表现 | 解决方案 |
|------|------|---------|
| 空串 vs NULL | Oracle 模式 ''=NULL, 迁移后 ''!='' | 统一用 NULL，或关闭空串转换 |
| 大小写 | Oracle 默认大写，MySQL 默认小写 | 统一用 lower_case_table_names=1 |
| 数据类型 | NUMBER -> 精度问题 | 使用 DECIMAL 而非 FLOAT |
| 序列/自增 | Oracle SEQUENCE, MySQL AUTO_INCREMENT | 迁移工具自动映射 |
| 存储过程 | PL/SQL 语法不兼容 | 改写为 MySQL 存储过程或应用层 |
| 分页 | ROWNUM vs LIMIT | 改写分页 SQL |
| 连接符 | `\|\|` vs CONCAT() | 统一用 CONCAT() 或应用层拼接 |
| 日期处理 | TO_DATE vs STR_TO_DATE | 改写日期函数 |
| MERGE | MERGE INTO vs INSERT ON DUPLICATE | 改写 UPSERT 语法 |
| 递归查询 | CONNECT BY vs WITH RECURSIVE | 改写递归 CTE |

**迁移策略:**

```
1. 评估: 用迁移评估工具扫描不兼容项
2. 改 DDL: 数据类型、索引、约束适配
3. 改 DML: SQL 语法适配
4. 改存储过程: 最大工作量，优先考虑移到应用层
5. 全量迁移: 数据迁移，校验
6. 增量同步: 迁移期间增量数据同步
7. 灰度切换: 逐步切流量验证
8. 回滚预案: 保留 Oracle 环境至少 1 个月
```
