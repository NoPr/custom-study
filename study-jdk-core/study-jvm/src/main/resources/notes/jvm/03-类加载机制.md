## 2. 双亲委派机制

```mermaid
sequenceDiagram
    participant App as Application CL
    participant Plat as Platform CL
    participant Boot as Bootstrap CL

    App->>App: findLoadedClass("com.example.User")<br/>→ 未找到
    App->>Plat: parent.loadClass("com.example.User")
    Plat->>Plat: findLoadedClass("com.example.User")<br/>→ 未找到
    Plat->>Boot: parent.loadClass("com.example.User")
    Boot->>Boot: findLoadedClass("com.example.User")<br/>→ 未找到
    Boot-->>Plat: ClassNotFoundException
    Plat->>Plat: findClass("com.example.User")<br/>→ 未找到
    Plat-->>App: ClassNotFoundException
    App->>App: findClass("com.example.User")<br/>→ classpath 中找到 → 加载！
    App-->>App: return Class<?>
```

**委托顺序**：Application → Platform → Bootstrap（自底向上）
**加载顺序**：Bootstrap → Platform → Application（自顶向下）

### loadClass 核心源码逻辑

```java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // ① 检查是否已经加载（全盘负责制）
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // ② 委托父加载器
                if (parent != null)
                    c = parent.loadClass(name, false);
                else
                    c = findBootstrapClassOrNull(name);
            } catch (ClassNotFoundException e) {
                // ③ 父加载器失败 → 自己加载
                c = findClass(name);
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}
```

## 3. 类加载过程（5 阶段）

```mermaid
flowchart TD
    Load["① 加载 Loading<br/>获取二进制字节流<br/>→ 方法区生成 Class 对象"]
    Verify["② 验证 Verification<br/>文件格式 / 元数据 / 字节码 / 符号引用验证"]
    Prepare["③ 准备 Preparation<br/>为 static 变量分配内存<br/>赋零值（非代码中的初始值）"]
    Resolve["④ 解析 Resolution<br/>符号引用 → 直接引用<br/>可与初始化交叉进行"]
    Init["⑤ 初始化 Initialization<br/>执行 static 块 + static 变量赋值<br/>这是真正执行 Java 代码的阶段"]

    Load --> Verify --> Prepare --> Resolve --> Init
```

**触发初始化的 6 种主动引用**：
1. new / getstatic / putstatic / invokestatic 指令
2. 反射调用（Class.forName）
3. 初始化子类前先初始化父类
4. 虚拟机启动时 main 方法所在类
5. MethodHandle 解析结果
6. default 接口方法实现类初始化

## 4. Class.forName vs loadClass

```mermaid
flowchart LR
    subgraph "Class.forName"
        FN1["加载"] --> FN2["连接"] --> FN3["初始化<br/>✅ 执行 static 块"]
    end
    subgraph "ClassLoader.loadClass"
        LC1["加载"] --> LC2["连接"] --> LC3["初始化<br/>❌ 不执行 static 块"]
    end
```

| 对比维度 | Class.forName | ClassLoader.loadClass |
|----------|---------------|----------------------|
| 初始化 | **执行** static 块 | **不执行** static 块 |
| 类加载器 | 默认用调用者的类加载器 | 必须手动指定 |
| 典型场景 | JDBC 驱动注册 | Spring IOC 懒加载 |
| 底层调用 | forName0(className, true, classLoader) | loadClass(name, false) |

### SPI 打破双亲委派流程

```mermaid
sequenceDiagram
    participant Code as 核心库代码<br/>(Bootstrap 加载)
    participant TCCL as Thread Context CL<br/>(默认 = AppCL)
    participant Impl as SPI 实现类<br/>(classpath)

    Code->>TCCL: Thread.currentThread()<br/>.getContextClassLoader()
    Note over Code,TCCL: 获取 AppCL
    Code->>TCCL: 读取 META-INF/services/xxx
    Code->>Impl: Class.forName(implName, true, AppCL)
    Note over Code,Impl: 用 AppCL 加载实现类<br/>打破了「Bootstrap 无法引用 AppCL 加载的类」的限制
```

### Tomcat WebAppClassLoader 加载顺序

```
① checkLoaded      → 已加载？返回
② isSystemClass    → J2SE 核心类？→ 委托 Bootstrap（不能打破）
③ WEB-INF/classes  → 自己加载
④ WEB-INF/lib      → 自己加载
⑤ 都没找到          → 委托父加载器 (Common → App → Bootstrap)
```
