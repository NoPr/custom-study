# 04 - NIO 面试高频问题

## Q1: BIO / NIO / AIO 的区别？

**答案：**

| 维度 | BIO | NIO | AIO |
|------|-----|-----|-----|
| 全称 | Blocking IO | Non-blocking IO | Asynchronous IO |
| 线程模型 | 1 连接 = 1 线程 | 1 线程管理 N 连接 | 回调通知 |
| 阻塞点 | accept/read/write | select() | 无阻塞 |
| 线程开销 | 极高 | 极低 | 低 |
| 编程复杂度 | 简单 | 中等 | 复杂 |
| 底层 | select/poll | epoll/kqueue | IOCP/epoll 模拟 |

**关键点：**
- BIO 适合连接数少、连接时长固定的场景
- NIO 适合高并发、短连接场景（如 IM、API 网关）
- AIO 适合连接数多、连接时间长的场景（如文件服务器），但 Netty 不推荐直接使用 JDK AIO

## Q2: NIO 的 Selector 是如何工作的？

**答案：**

Selector 是多路复用器的 Java 封装，单线程通过它管理多个 Channel。

**工作流程：**
1. `Selector.open()` → 创建 Selector（底层根据 OS 选择 EPoll/KQueue/WindowsSelector）
2. `ServerSocketChannel.register(selector, OP_ACCEPT)` → 将 Channel 注册到 Selector
3. `selector.select()` → 阻塞等待直到有就绪事件
4. 遍历 `selector.selectedKeys()`，根据事件类型分发处理
5. **手动 `iterator.remove()`** 移除已处理的 key
6. 回到步骤 3

**关键细节：**
- `select()` 阻塞，`select(long timeout)` 超时阻塞，`selectNow()` 非阻塞
- Selector 不会自动清除 selectedKeys，必须手动 remove，否则下次 select 会重复处理
- `wakeup()` 可以唤醒阻塞在 select() 上的线程

## Q3: epoll 和 select/poll 的区别？

**答案：**

| | select | poll | epoll |
|---|--------|------|-------|
| fd 存储 | 数组（bitmap） | 链表 | 红黑树 + 就绪链表 |
| 获取就绪事件 | O(n) 遍历所有 fd | O(n) 遍历所有 fd | O(1) 直接取就绪链表 |
| fd 数量限制 | 1024（FD_SETSIZE）| 无上限 | 无上限 |
| 内核/用户空间拷贝 | 每次拷贝完整 fd_set | 每次拷贝 pollfd 结构体 | epoll_wait 只拷贝就绪 fd |
| fd 增删效率 | O(n) | O(n) | O(log n)（红黑树） |

**核心差异：**
- select/poll 每次调用都需要把全部 fd 从用户空间拷贝到内核空间
- epoll 通过 `epoll_ctl` 注册 fd，内核用红黑树维护，`epoll_wait` 只返回就绪 fd
- epoll 支持 ET（边缘触发）和 LT（水平触发）两种模式

## Q4: ByteBuffer 的 flip、clear、compact 方法的作用？

**答案：**

```
初始状态（容量 10）:
  pos=0  lim=10  cap=10  [ _ _ _ _ _ _ _ _ _ _ ]

put("HELLO") 写入 5 字节:
  pos=5  lim=10  cap=10  [ H E L L O _ _ _ _ _ ]

flip() 写 → 读:
  pos=0  lim=5   cap=10  limit=position, position=0

get() 读了 2 字节 (H, E):
  pos=2  lim=5   cap=10

compact() 压缩未读数据:
  [ L L O L O _ _ _ _ _ ]  未读的 LLO 移到头部
  pos=3  lim=10  cap=10    position=未读字节数

clear() 读 → 写:
  pos=0  lim=10  cap=10    数据不清零，仅重置指针
```

**关键区别：**
- `flip()`：写模式 → 读模式，保留实际写入的数据范围
- `clear()`：切换到写模式，不擦除数据（仅重置指针）
- `compact()`：保留未读数据到头部，position 指向未读数据之后（适合半包场景）
- `rewind()`：重新读，position=0，不改变 limit

## Q5: 什么是零拷贝？Java 中如何实现？

**答案：**

**定义：** 零拷贝不是零次拷贝，而是「减少不必要的 CPU 拷贝，让 CPU 从数据搬运中解放」。

**三种方案对比：**

| 方案 | CPU 拷贝 | DMA 拷贝 | 上下文切换 |
|------|---------|---------|-----------|
| 传统 read/write | 2 | 2 | 2 |
| mmap + write | 1 | 2 | 2 |
| sendfile | 0 | 2 | 2 |

**Java 实现：**
1. `FileChannel.map()` → MappedByteBuffer（mmap）
2. `FileChannel.transferTo()` → 底层调用 sendfile64（sendfile）

**场景：**
- Kafka：日志段 mmap，消费 transferTo
- Netty：DefaultFileRegion 封装 transferTo

## Q6: 粘包/半包是什么？如何解决？

**答案：**

**产生原因：**
1. Nagle 算法：将小包合并发送
2. MSS 限制：超过 1460 字节被 TCP 分片
3. 应用层读写速度不匹配

**三种解决方案：**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 定长解码 | 每条消息固定 N 字节 | Modbus、固定指令协议 |
| 分隔符解码 | 特殊字符分割（\n、\r\n）| Redis、HTTP 头 |
| LengthField | 消息头 + 消息体，头记录长度 | Dubbo、gRPC、MQTT |

**Netty 内置解码器：**
- `FixedLengthFrameDecoder`
- `DelimiterBasedFrameDecoder`
- `LengthFieldBasedFrameDecoder`

## Q7: DirectByteBuffer 和 HeapByteBuffer 的区别？

**答案：**

| | HeapByteBuffer | DirectByteBuffer |
|---|---------------|------------------|
| 内存位置 | JVM 堆内 | 堆外（Direct Memory） |
| 分配速度 | 快（TLAB 分配） | 慢（系统调用 malloc） |
| IO 操作 | 多一次拷贝（堆→堆外） | 零拷贝（直接 DMA） |
| GC 影响 | 受 GC 管理，可能移动 | 不受 GC 移动影响 |
| 回收 | GC 自动回收 | Cleaner 虚引用回收 |

**选择建议：**
- IO 密集型（网络/文件读写）→ DirectByteBuffer
- 业务计算密集型 → HeapByteBuffer
- 长生命周期的大 buffer → DirectByteBuffer

## Q8: 为什么 Netty 选择 NIO 而不是 AIO？

**答案：**

1. **跨平台不一致**：Java AIO 在 Linux 上用 epoll + 线程池模拟，Windows 才是真正的 IOCP。epoll 直接使用性能更好
2. **生态成熟**：epoll 是 Linux 内核级支持，十几年优化积累
3. **Netty 已在 NIO 上建成完整体系**：EventLoopGroup、Pipeline、ChannelHandler 等，切换 AIO 收益不明显
4. **清理问题**：AIO 异步写未完成就关闭 Channel 可能丢数据，NIO 更可控

## Q9: NIO 编程中的常见陷阱？

**答案：**

1. **Selector.selectedKeys() 未 remove**：导致重复处理、CPU 100%
2. **OP_WRITE 滥用**：大部分时间通道可写，一直注册导致空转
3. **ByteBuffer 未 flip**：读完直接写，数据不对
4. **忽略 read() 返回 -1**：客户端断开未关闭 Channel
5. **非阻塞 write() 未完全写入**：ByteBuffer.hasRemaining() 仍需继续写
6. **Selector 多线程竞争**：多个线程同时 select 同一 Selector 会阻塞
