## Q1: WebSocket 握手原理是怎样的？和 HTTP 长轮询有什么区别？

**握手过程（4 步）：**

1. 客户端发起 HTTP GET，携带 `Upgrade: websocket` + `Connection: Upgrade` + `Sec-WebSocket-Key`
2. 服务端计算 `Sec-WebSocket-Accept = Base64(SHA1(clientKey + MAGIC_GUID))`
3. 服务端返回 `101 Switching Protocols`
4. TCP 连接升级为 WebSocket 全双工通道

**与 HTTP 长轮询对比：**

| 维度 | HTTP 长轮询 | WebSocket |
|------|-----------|-----------|
| 通信模式 | 半双工（请求-响应） | 全双工 |
| 服务端推送 | 被动（等客户端请求） | 主动推送 |
| 帧头开销 | 每次 500-800 字节 HTTP 头 | 2-10 字节 |
| 连接数 | 每次请求/响应一个连接 | 一个持久连接 |
| 浏览器支持 | 所有 | IE10+ |

## Q2: 如何设计心跳保活和断线重连方案？

**心跳（两种方案组合）：**

```
方案 A：客户端 Ping-Pong（主动心跳）
  客户端 30s → Ping(0x9) → 服务端 → Pong(0xA) → 客户端
  连续 3 次未收到 Pong → 判定断线

方案 B：服务端 IdleStateHandler（被动检测）
  readerIdle=15s → 发 Ping 探测
  writerIdle=30s → 发 Pong 保活
  allIdle=60s   → 关闭连接
```

**重连（指数退避 + Jitter）：**

```
间隔序列: 1s → 2s → 4s → 8s → 16s → 30s(max)
加 ±25% 随机 Jitter 防止惊群

状态机:
  CONNECTED → DISCONNECTED → RECONNECTING → RECONNECTED(成功)
                                           → FAILED(超过最大重试)

Session 恢复:
  重连成功 → sessionId 不变 → userId 重新绑定 → 补推离线消息
```

## Q3: 10 万级 WebSocket 连接如何做内存和性能优化？

**内存优化（5 个方向）：**

| 优化项 | 具体措施 |
|-------|---------|
| Session 瘦身 | UUID→短ID(8B)，包装类→基本类型，延迟初始化 attributes |
| Netty ByteBuf | PooledByteBufAllocator + DirectByteBuf（堆外内存） |
| TCP 缓冲区 | SO_RCVBUF/SO_SNDBUF 从 128KB 降至 8KB |
| 序列化换出 | 非活跃 Session 序列化到 Redis，用时反序列化 |
| 压缩 | Permessage-deflate 扩展（RFC 7692），消息体压缩 |

**性能优化：**

| 优化项 | 具体措施 |
|-------|---------|
| 操作系统 | `ulimit -n 655350`，调整 `tcp_tw_reuse`、`tcp_fin_timeout` |
| Netty 线程 | Boss 1 个 + Worker = CPU 核数 |
| 无锁化 | ConcurrentHashMap、AtomicLong 替代 synchronized |
| 零拷贝 | FileRegion.transferTo() 发送大文件 |

**单 Session 内存对比：**

```
优化前: UUID(36B) + String userId(20B) + Map attributes(200B+) ≈ 300B
优化后: shortId(8B) + primitive fields + lazy Map ≈ 100B
10w 连接: 30MB → 10MB
```

## Q4: WebSocket 集群方案？如何跨节点推送消息？

**三种方案对比：**

```
方案 1: Redis Pub/Sub（轻量场景推荐）
┌──────┐  PUBLISH   ┌───────┐  SUBSCRIBE   ┌──────┐
│ 节点A │ ────────▶ │ Redis │ ◀──────── │ 节点B │
└──────┘            └───────┘            └──────┘
                                            │
                                    持有 uid=123 的 Session
                                            │
                                        推送消息
  优点: 简单低延迟(~1ms)
  缺点: 全广播，非目标节点也收到


方案 2: Kafka/RocketMQ（高吞吐场景）
  消息 → MQ Topic(按userId分区) → 各节点 Consumer 消费
  优点: 可靠可回溯，消息不丢
  缺点: 延迟略高(~10ms)，运维复杂


方案 3: 一致性哈希路由（最简洁）
  Nginx/API网关 → userId一致性哈希 → 固定节点 → 无需跨节点转发
  优点: 架构最简单，零额外开销
  缺点: 节点上下线需重新分配连接
```

**生产推荐组合**：一致性哈希路由 + Redis Pub/Sub 兜底（节点变更时临时转发）。

## Q5: WebSocket 和 Socket 有什么区别？各自适用场景？

| 比较维度 | WebSocket | Socket (TCP) |
|---------|-----------|-------------|
| OSI 层级 | 应用层（RFC 6455） | 传输层（操作系统 API） |
| 协议规范 | 标准帧格式、握手流程 | 无规范，自定义通信协议 |
| 浏览器支持 | 原生 JS `new WebSocket()` | 不支持 |
| 防火墙穿透 | HTTP Upgrade，天然兼容 | 需额外配置 |
| 帧头开销 | 2-10 字节 | 无（自定义） |
| 安全 | `wss://` (TLS) | SSL/TLS |
| 典型场景 | Web 推送、IM、协同编辑 | 游戏、IoT、自定义协议 |

**核心关系**：WebSocket 本质是 TCP 的上层封装：`WebSocket = TCP + HTTP Upgrade + 标准帧协议`。

**选型决策树**：

```
需要浏览器/Js 直接调用？ ──是──▶ WebSocket
         │
         否
         │
         ▼
需要标准协议？ ──是──▶ WebSocket
         │
         否
         │
         ▼
    自定义协议 → Socket (TCP/UDP)
```

## 补充：Nginx 代理 WebSocket 配置

```nginx
location /ws/ {
    proxy_pass http://backend;
    proxy_http_version 1.1;                    # ★ 必须是 1.1
    proxy_set_header Upgrade $http_upgrade;     # ★ 透传 Upgrade
    proxy_set_header Connection "upgrade";      # ★ 透传 Connection
    proxy_read_timeout 3600s;                   # 长连接超时
    proxy_send_timeout 3600s;
    proxy_buffering off;                        # 禁用缓冲（实时性）
}
```

**三个必须**：
1. `proxy_http_version 1.1` -- HTTP/1.0 不支持 Upgrade
2. `proxy_set_header Upgrade $http_upgrade` -- 透传升级头
3. `proxy_set_header Connection "upgrade"` -- 透传连接方式
