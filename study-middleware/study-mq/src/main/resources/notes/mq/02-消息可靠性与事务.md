# 02-消息可靠性与事务

## 消息丢失三大场景

```mermaid
graph TD
    subgraph 场景1["场景 1: Producer → Broker"]
        P1["Producer"] -->|"1.发送消息"| NET["🌐 网络"]
        NET -->|"2.网络抖动"| LOSS1["❌ 消息丢失"]
        P1 -->|"3.重试"| NET2["🌐 网络"]
        NET2 -->|"4.重试成功"| B1["Broker ✅"]
    end

    subgraph 场景2["场景 2: Broker → Consumer"]
        B2["Broker"] -->|"1.消息在内存"| RAM["内存 (PageCache)"]
        RAM -->|"2.Broker 宕机"| LOSS2["❌ 异步刷盘消息丢失"]
        B2 -->|"3.同步刷盘"| DISK["磁盘 ✅"]
    end

    subgraph 场景3["场景 3: Consumer → 业务"]
        C3["Consumer"] -->|"1.消费成功"| BIZ["业务处理"]
        BIZ -->|"2.DB 写入失败"| RETRY["重试消费"]
        RETRY -->|"3.幂等检查"| IDEMP["msgId 去重 ✅"]
    end
```

### 场景 1: Producer → Broker

**问题**: 网络抖动导致消息未到达 Broker
**方案**:
- 同步发送 + 重试机制 (retryTimesWhenSendFailed=3)
- 故障延迟机制 (sendLatencyFaultEnable=true)：记录 Broker 延迟，避开慢节点

### 场景 2: Broker → Consumer

**问题**: Broker 宕机，内存中未刷盘消息丢失
**方案**:
- 同步刷盘 (flushDiskType=SYNC_FLUSH)：每条消息 fsync 到磁盘
- 同步复制 (brokerRole=SYNC_MASTER)：Slave 确认后才返回成功
- DLedger (Raft) 自动主备切换

### 场景 3: Consumer → 业务

**问题**: 消费成功但业务处理失败
**方案**:
- 手动 ACK (CONSUME_SUCCESS / RECONSUME_LATER)
- 业务幂等：基于 msgId 或业务唯一键去重
- 死信队列：重试 16 次后进入 DLQ

## 事务消息两阶段提交

```mermaid
sequenceDiagram
    participant P as Producer
    participant B as Broker
    participant DB as 本地数据库
    participant C as Consumer

    Note over P,C: 阶段1 - 发送半消息
    P->>B: sendMessageInTransaction (半消息)
    B-->>P: OK (半消息存储, 消费者不可见)
    Note over B: 半消息标记为 TRAN_MSG

    Note over P,C: 阶段2 - 执行本地事务
    P->>DB: 执行本地事务 (写订单表)
    DB-->>P: 事务结果 (成功/失败)

    alt 本地事务成功
        P->>B: commit
        Note over B: 半消息 → 正常消息 (消费者可见)
        B-->>C: 消息可消费
    else 本地事务失败
        P->>B: rollback
        Note over B: 半消息删除
    else Producer 宕机 (超时)
        Note over B: 半消息超时, 触发回查
        B->>P: checkLocalTransaction(txId)
        P->>DB: 查询本地事务状态
        DB-->>P: 事务结果
        P-->>B: COMMIT / ROLLBACK
    end
```

## ACK 重试流程

```mermaid
graph TD
    CONSUME["消费消息"] --> PROCESS["业务处理"]
    PROCESS -->|"成功"| SUCCESS["返回 CONSUME_SUCCESS"]
    PROCESS -->|"失败"| RETRY_CHECK{"重试次数 < 16?"}
    RETRY_CHECK -->|"是"| DELAY["延迟重试<br/>10s→30s→1m→2m→...→2h"]
    DELAY --> CONSUME
    RETRY_CHECK -->|"否"| DLQ["进入死信队列 (DLQ)"]
    SUCCESS --> ACK["更新 offset"]
    DLQ --> ALERT["人工处理/告警"]
```

### 重试时间间隔（RocketMQ 默认）

| 重试次数 | 间隔 | 说明 |
|---------|------|------|
| 1-3 | 10s | 快速失败，可能是瞬时异常 |
| 4-6 | 30s | 中等等待 |
| 7-9 | 1min | 可能是服务压力 |
| 10-12 | 10min | 较长恢复时间 |
| 13-15 | 30min | 需要人工关注 |
| 16+ | 2h | 进入 DLQ |

## 消息可靠性保障层级

```
Level 1 (最低): ASYNC_FLUSH + ASYNC_MASTER
  → 适用: 日志采集, 可容忍少量丢失

Level 2: ASYNC_FLUSH + SYNC_MASTER
  → 适用: 普通业务消息, Slave 有备份

Level 3: SYNC_FLUSH + SYNC_MASTER
  → 适用: 金融/支付, 零丢失要求

Level 4 (最高): SYNC_FLUSH + SYNC_MASTER + DLedger
  → 适用: 银行核心交易, 自动故障切换
```

## 关键面试点

1. **消息可靠性的三个维度**: 生产端(重试) + Broker端(刷盘/复制) + 消费端(手动ACK+幂等)
2. **事务消息为什么不需要 XA/2PC**: 通过半消息+回查实现最终一致，比 XA 轻量且性能好
3. **幂等消费的三种实现**: 数据库唯一索引、Redis setnx、本地消息表
4. **同步刷盘 vs 同步复制哪个开销大**: 同步复制开销大（网络），因为磁盘顺序写远快于网络传输
