# 04-面试高频问题

## Q1: 如何保证消息不丢失？

**回答框架**: 从三个环节分析 —— 生产端、Broker端、消费端

```
1. 生产端 (Producer):
   - 同步发送 (send(msg) 而非 sendOneway)
   - 重试机制: retryTimesWhenSendFailed=3
   - 故障延迟: sendLatencyFaultEnable=true

2. Broker端:
   RocketMQ: 同步刷盘(SYNC_FLUSH) + 同步复制(SYNC_MASTER)
   Kafka: acks=all + min.insync.replicas=2 + replication.factor=3
   → RocketMQ 同步刷盘开销 < 同步复制开销
   → Kafka 的 min.insync.replicas=2 确保至少 2 个副本

3. 消费端 (Consumer):
   - 手动 ACK (CONSUME_SUCCESS 才提交 offset)
   - 业务幂等: 基于 msgId/业务唯一键
   - 死信队列: 重试 16 次后进入 DLQ 人工处理

总结: "两端一存，三重保障"
```

## Q2: 如何保证消息不重复消费？

**核心**: 消息队列保证 at-least-once，业务端保证幂等

```
幂等方案 (按推荐度排序):

1. 数据库唯一索引 (最强):
   INSERT INTO consumed_msg (msg_id) VALUES (?)
   → 依赖 UNIQUE KEY 保证不重复

2. Redis setnx (高性能):
   SETNX msg:order:1001 "1" EX 86400
   → TTL 根据消息有效期设置

3. 本地消息表 (无需外部依赖):
   SELECT FROM consumed WHERE msg_id = ?
   INSERT + UPDATE 在同一事务

4. 版本号机制 (乐观锁):
   UPDATE order SET status='PAID', version=version+1
   WHERE id=? AND version=?

5. 业务自然幂等:
   - 金额设为绝对值: SET balance=100 (非 increment)
   - 状态机: 已支付 → 不再处理支付消息
```

## Q3: 如何保证消息顺序？

**分类讨论**:

```
1. 分区有序 (最常用):
   RocketMQ: MessageQueueSelector + (queues,msg,arg) → queueId
   Kafka: 按 key hash 分配到固定 Partition
   → 相同业务 ID (如 orderId) 发到同一 Queue/Partition

2. 全局有序 (特殊场景):
   RocketMQ: Topic 仅 1 个 WriteQueue
   Kafka: Topic 仅 1 个 Partition
   → 禁用负载均衡, 单线程消费
   → 缺点: 性能瓶颈

3. 顺序中断处理:
   - 重平衡: Consumer 宕机 → Queue 重新分配 → 新 Consumer 进度不连续
   - 消费失败: SUSPEND_CURRENT_QUEUE_A_MOMENT → 兄弟 Queue 继续消费 → 整体乱序
   → 方案: 业务侧加 seq 字段排序/丢弃过期消息
```

## Q4: 消息积压怎么处理？

**排查 → 临时扩容 → 长期方案**:

```
1. 排查原因:
   - Consumer 挂了? → 重启
   - Consumer 消费慢? → 优化代码
   - 流量突增? → 临时扩容

2. 临时方案 (快速消积):
   step1: 新建 Topic (如 OrderTopic-Retry, 队列数 x10)
   step2: 临时 Consumer 转发原 Topic → 新 Topic (不做业务)
   step3: 启动 10x Consumer 消费新 Topic → 快速消积

3. 长期方案:
   - 增加 Queue/Partition 数量 (需预先规划)
   - Consumer 批量消费 + 批量 ACK
   - 异步处理 + 线程池
   - 限流 (生产端)
```

## Q5: RocketMQ 事务消息原理？

```
两阶段 + 回查:

阶段 1: 发送半消息 (Half Message)
  Producer → Broker: 发送 PREPARE 消息
  Broker: 存储半消息, 标记为 TRAN_MSG (消费者不可见)

阶段 2: 执行本地事务
  Producer: 执行本地事务 (如 DB 写操作)
  成功 → commit (半消息 → 正常消息, 消费者可见)
  失败 → rollback (半消息删除)

阶段 3: 事务状态回查 (CheckListener)
  触发时机: Broker 在指定时间内未收到 commit/rollback
  Broker → Producer.CheckListener.checkLocalTransaction()
  Producer: 查询本地事务状态 → 返回 commit/rollback/unknown

对比 XA/2PC:
  XA: 全局锁, 强一致, 性能差
  RocketMQ: 半消息 + 回查, 最终一致, 性能好

不在 XA 范畴内的原因:
  - 第一阶段不持有锁 (半消息只是不可见)
  - 第二阶段异步 (没有 prepare→commit 阻塞)
  - Broker 主动回查 (不需要 TM 协调)
```

## Q6: RocketMQ vs Kafka vs JPA/Hibernate

| 维度 | RocketMQ | Kafka | JPA/Hibernate |
|------|----------|-------|---------------|
| 定位 | 消息队列 | 流平台 | ORM 框架 |
| 核心功能 | 消息收发 | 消息+流处理 | DB 映射+缓存 |
| 注册中心 | NameServer (自研) | ZK/KRaft | 无 |
| 存储 | CommitLog | Partition 日志 | DB 表 |
| 事务 | 半消息+回查 | PID+Coordinator | JTA/XA |
| 事务消息 | **原生支持** | KIP-98 | 不支持 |
| 顺序消息 | **原生分区+全局** | Partition 内 | 不支持 |
| 延迟消息 | **18 级延迟** | 无内置 | 不支持 |
| 消息过滤 | **Tag+SQL92** | 消费端处理 | JPQL/SQL |
| 适用场景 | **业务消息**(订单/支付) | **日志+流处理** | **数据库操作** |
| 性能 | ~10w TPS | ~100w TPS | DB 瓶颈 |
| 学习成本 | 中 | 中 | 高 |
| 运维复杂度 | 低 (无外部依赖) | 高 (需 ZK) | 低 |
| 阿里生态 | **深度集成** | 一般 | 一般 |
| SpringBoot | rocketmq-spring | spring-kafka | spring-data-jpa |

**一句话选型 (面试金句)**:
> RocketMQ 做业务消息，Kafka 做流处理，不冲突可以都上。
