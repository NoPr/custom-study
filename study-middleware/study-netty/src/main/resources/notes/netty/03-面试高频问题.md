# 面试高频问题

> Netty 面试 5 题自测，覆盖 Reactor 模型、零拷贝、Pipeline、粘包半包和高性能设计。

---

## 问题 1：Netty 高性能的原因有哪些？

**回答框架**：I/O 模型 + 线程模型 + 零拷贝 + 无锁化 + 池化 + 并发优化

### 六大原因

| 序号 | 维度 | 具体机制 |
|------|------|----------|
| 1 | **I/O 模型** | NIO 多路复用，一个线程管理多个连接，epoll 边缘触发 |
| 2 | **线程模型** | 主从 Reactor，Boss 只管 accept，Worker 管 read/write |
| 3 | **零拷贝** | sendfile + CompositeByteBuf + Slice + DirectBuffer |
| 4 | **无锁串行化** | Channel 绑定固定 EventLoop，同线程操作无锁 |
| 5 | **内存池化** | PooledByteBufAllocator，jemalloc 风格 Arena 算法 |
| 6 | **并发优化** | MPSC 队列 + FastThreadLocal + HashedWheelTimer |

### 扩展回答（加分项）

- **MPSC 队列**：多生产者单消费者，Netty 外部线程提交任务到 EventLoop
- **FastThreadLocal**：使用数组索引代替 JDK 的线性探测，快 3 倍
- **时间轮（HashedWheelTimer）**：O(1) 复杂度管理定时任务，替代 `ScheduledExecutorService`
- **对象池（Recycler）**：轻量级对象池，复用 Handler 上下文等短生命周期对象

## 问题 2：Netty 的 Reactor 模型是怎样的？

### 三层模型

```
MainReactor (Boss Group)
    │
    │ accept 新连接
    │ 轮询注册
    ▼
SubReactor (Worker Group)
    │
    │ read/write + 业务处理
    ▼
Pipeline (Handler 链)
```

### 核心要点

1. **Boss Group**：通常 1 个线程，监听 `ServerSocketChannel` 的 `OP_ACCEPT`
2. **Worker Group**：CPU 核数 x 2 个线程，处理 `SocketChannel` 的 `OP_READ`/`OP_WRITE`
3. **连接绑定**：每个 `Channel` 从创建到关闭始终绑定同一个 `EventLoop` 线程
4. **无锁化**：同一 Channel 的所有 Handler 操作都在同一线程，无需同步

### 与其他框架对比

| 框架 | 模型 | 特点 |
|------|------|------|
| Netty | 主从 Reactor | accept 与 IO 分离，适合高并发 |
| Redis 6.0 前 | 单 Reactor 单线程 | 简单，但 CPU 密集型操作会阻塞 IO |
| Tomcat（BIO） | 一连接一线程 | 200 连接即频繁上下文切换 |
| Nginx | 多进程 + epoll | 每个 Worker 独立 Reactor |

## 问题 3：Pipeline 责任链是如何工作的？

### 数据结构

```
Pipeline = 双向链表
Head ↔ Handler1 ↔ Handler2 ↔ ... ↔ Tail
```

### 事件传播

| 事件类型 | 传播方向 | 触发方法 |
|----------|----------|----------|
| Inbound（读入） | Head → Tail | `ctx.fireChannelRead(msg)` |
| Outbound（写出） | Tail → Head | `ctx.write(msg)` |
| 异常 | 当前 → Tail | `ctx.fireExceptionCaught(cause)` |

### 关键设计

1. **每个 Handler 被包装成 ChannelHandlerContext**，持有 prev/next 指针
2. **Handler 可以选择中断传播**：不调用 `ctx.fireXxx()` 即可停止
3. **Outbound 是"找 prev 方向"**，与直觉相反，容易混淆
4. **异常逐节点向后查找**，找到第一个不传播的 ExceptionHandler

### ChannelHandlerContext vs Channel vs ChannelPipeline

| 对象 | 生命周期 | 用途 |
|------|----------|------|
| ChannelPipeline | 与 Channel 同生命周期 | 管理 Handler 链 |
| ChannelHandler | 可共享（@Sharable） | 处理 IO 事件 |
| ChannelHandlerContext | 每次添加 Handler 创建 | Handler 与 Pipeline 的桥梁 |
| Channel | 每个连接一个 | 代表一个 Socket 连接 |

## 问题 4：Netty 的零拷贝具体有哪些？

### 四层零拷贝

```
┌─────────────────────────────────────────┐
│ ④ 堆外内存：DirectBuffer 避免堆内→堆外拷贝   │
├─────────────────────────────────────────┤
│ ③ 切片共享：ByteBuf.slice() 共享同块内存     │
├─────────────────────────────────────────┤
│ ② 逻辑合并：CompositeByteBuf 零拷贝组合      │
├─────────────────────────────────────────┤
│ ① 系统调用：sendfile(transferTo) 不经过用户态 │
└─────────────────────────────────────────┘
```

### 各层详解

| 层 | API | 原理 | 典型场景 |
|----|-----|------|----------|
| sendfile | `FileRegion` | DMA gather，内核态直接传输 | 文件服务器 |
| CompositeByteBuf | `Unpooled.wrappedBuffer()` | 记录引用数组，不 copy | HTTP Header+Body |
| Slice | `ByteBuf.slice()` | 共享内存 + 独立读写指针 | 协议头/体分离 |
| DirectBuffer | `PooledDirectByteBuf` | 堆外分配，IO 直达 | 所有网络 IO |

### 与 Kafka 零拷贝对比

| 维度 | Netty | Kafka |
|------|-------|-------|
| 发送文件到网络 | `transferTo` (sendfile) | `transferTo` (sendfile) |
| 写日志到磁盘 | N/A | `mmap` (PageCache) |
| 读日志发消费者 | N/A | `sendfile` (PageCache → Socket) |
| 应用层零拷贝 | CompositeByteBuf, Slice | `MessageSet` 零拷贝传输 |

## 问题 5：粘包半包怎么解决？

### 原因一句话

> TCP 是面向字节流的协议，不保留应用层消息边界。

### 根本原因

| 原因 | 说明 |
|------|------|
| Nagle 算法 | 合并小的 TCP 包，减少网络开销（默认开启） |
| MSS 分段 | 超过 1460 字节时 IP 层分片 |
| 滑动窗口 | 接收缓冲区满时阻塞，恢复后一起到达 |
| 应用层读写缓冲区 | 发送端批量写，接收端缓冲区大小有限 |

### 五种解决方案

| 方案 | 原理 | 代码 | 推荐度 |
|------|------|------|--------|
| LineBasedFrameDecoder | 换行符分隔 | `new LineBasedFrameDecoder(1024)` | 文本协议用 |
| DelimiterBasedFrameDecoder | 自定义分隔符 | `new DelimiterBasedFrameDecoder(1024, delimiter)` | 简单自定义协议 |
| FixedLengthFrameDecoder | 固定长度 | `new FixedLengthFrameDecoder(20)` | 定长协议 |
| LengthFieldBasedFrameDecoder | 长度字段 | `new LengthFieldBasedFrameDecoder(65536, 0, 4, 0, 4)` | 通用，最推荐 |
| 自定义解码器 | 完全自定义 | 继承 `ByteToMessageDecoder` | 复杂协议 |

### 面试最佳回答模板

> "粘包半包本质是 TCP 字节流无边界导致的。Netty 提供了多种 FrameDecoder 来解决：
> 最常用的是 `LengthFieldBasedFrameDecoder`，协议头带魔数和长度字段；
> 文本协议用 `LineBasedFrameDecoder` 按换行符分割；
> 定长协议用 `FixedLengthFrameDecoder`。
> 生产环境推荐魔数 `+` 长度字段 `+` 自定义协议，Encoder/Decoder 成对使用。"
