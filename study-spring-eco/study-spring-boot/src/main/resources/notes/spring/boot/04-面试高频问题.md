# Spring Boot 面试高频问题

## Q1：Spring Boot 自动配置原理？

**回答框架：**

Spring Boot 通过 `@SpringBootApplication` 启动，该注解包含 `@EnableAutoConfiguration`。
`@EnableAutoConfiguration` 通过 `@Import` 导入 `AutoConfigurationImportSelector`。

该选择器在 `selectImports()` 方法中执行以下步骤：
1. 读取 `META-INF/spring.factories` 或 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件
2. 获取所有候选自动配置类（约 180 个）
3. 根据条件注解（`@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等）过滤
4. 返回满足条件的配置类列表
5. Spring 容器处理这些配置类，注册 Bean

**一句话：** Spring Boot 通过 SPI 机制加载所有自动配置类，根据类路径、Bean 存在性等条件过滤后，按需注册 Bean。

**关键类：** `AutoConfigurationImportSelector`、`SpringFactoriesLoader`、`ImportCandidates`

## Q2：如何自定义一个 Spring Boot Starter？

**回答框架：**

自定义 Starter 包含三个核心部分：

1. **属性类：** 使用 `@ConfigurationProperties(prefix = "xxx")` 定义，提供默认值和 getter/setter。

```java
@ConfigurationProperties(prefix = "custom")
public class CustomProperties {
    private String host = "localhost";
    private int port = 8080;
    // getters + setters
}
```

2. **自动配置类：** 使用 `@Configuration` + `@EnableConfigurationProperties`，
   配合 `@ConditionalOnClass` / `@ConditionalOnMissingBean` 确保健壮性。

```java
@Configuration
@EnableConfigurationProperties(CustomProperties.class)
@ConditionalOnClass(CustomService.class)
public class CustomAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public CustomService customService(CustomProperties properties) {
        return new CustomService(properties);
    }
}
```

3. **注册 SPI：** 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 中添加自动配置类的全限定名。

```
com.example.custom.CustomAutoConfiguration
```

**命名规范：** 官方 Starter 用 `spring-boot-starter-*`，第三方 Starter 用 `*-spring-boot-starter`。

## Q3：@ConditionalOnMissingBean 的作用？

**回答：**

`@ConditionalOnMissingBean` 的条件是：**只有当容器中不存在指定 Bean 时，才注册当前 Bean**。

**核心价值：允许用户覆盖默认配置。**

### 典型场景

| 场景 | 默认行为 | 用户自定义 |
|------|----------|------------|
| DataSource 配置 | Spring Boot 自动创建 `DataSource` Bean | 用户自定义 `DataSource` 后，默认的不会创建 |
| RedisTemplate | 自动创建默认 `RedisTemplate` | 用户自定义 `RedisTemplate` Bean 后，默认的不会创建 |
| ObjectMapper | 自动创建默认 `ObjectMapper` | 用户自定义 `ObjectMapper` 后，默认的不会创建 |

### 工作原理

容器在处理 `@Bean` 方法时，先检查容器中是否已有匹配的 Bean：
- 有 → 跳过当前 `@Bean` 方法
- 无 → 执行 `@Bean` 方法，注册当前 Bean

## Q4：spring.factories vs .imports 文件的区别？

**回答：**

| 对比维度 | spring.factories | .imports 文件 |
|----------|:-----------------:|:-------------:|
| 引入版本 | Spring Boot 1.x | Spring Boot 2.7+ |
| 文件路径 | `META-INF/spring.factories` | `META-INF/spring/...AutoConfiguration.imports` |
| 格式 | `key=value1,\\nvalue2` | 每行一个全限定类名 |
| 用途 | 通用 SPI（多种 key） | 专用于自动配置 |
| 加载方式 | `SpringFactoriesLoader.loadFactoryNames()` | `ImportCandidates.load()` |
| key 冲突风险 | 高（单文件多用途） | 无（职责专一） |
| Spring Boot 3.x | 仍支持，向后兼容 | **推荐使用** |

**建议：**
- 新项目: 使用 `.imports` 文件
- 旧项目: 逐步迁移
- 两种格式 Spring Boot 3.x 都支持

## Q5：Spring Boot 与 Spring 的区别？

**回答：**

| 对比维度 | Spring Framework | Spring Boot |
|----------|:-----------------:|:-----------:|
| 定位 | IoC/DI 框架 | Spring 之上的一站式脚手架 |
| 配置方式 | XML / Java Config | **约定大于配置，开箱即用** |
| 依赖管理 | 手动管理版本 | starter 统一管理版本 |
| 内嵌容器 | 需要外部容器 | **内嵌 Tomcat/Jetty/Undertow** |
| 自动配置 | 需手动配置 | **基于条件的自动配置** |
| Actuator | 无 | **内置监控端点**（`/actuator`） |
| 启动类 | 无 | `@SpringBootApplication` |
| 部署 | WAR 包部署 | FAT JAR 独立运行 |

**核心区别：**

1. **自动配置：** Spring Boot 根据依赖自动配置 Bean，Spring 需要手动配置
2. **内嵌容器：** Spring Boot 内嵌 Tomcat，Spring 依赖外部容器
3. **依赖管理：** Spring Boot starter 统一版本管理，Spring 需手动管理
4. **开箱即用：** Spring Boot 约定大于配置，Spring 需大量配置

## Q6：如何排除不需要的自动配置？

**三种方式：**

### 方式一：@SpringBootApplication 直接排除

```java
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

### 方式二：application.yml 配置

```yaml
spring:
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
      - org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration
```

### 方式三：@EnableAutoConfiguration 排除

```java
@Configuration
@EnableAutoConfiguration(exclude = {RedisAutoConfiguration.class, AopAutoConfiguration.class})
public class MyConfig {
}
```

### 常见排除场景

| 排除的配置类 | 原因 |
|-------------|------|
| `DataSourceAutoConfiguration` | 多数据源、JNDI 数据源、手动 JdbcTemplate |
| `RedisAutoConfiguration` | 自定义 Redis 客户端、不使用 Redis |
| `AopAutoConfiguration` | 不需要 AOP 代理 |
| `SecurityAutoConfiguration` | 自定义安全配置、无安全需求 |
| `ThymeleafAutoConfiguration` | 使用其他模板引擎、纯 REST API |

### 如何排查 10w+ 连接的优化？

| 优化方向 | 具体措施 |
|----------|----------|
| 系统参数 | `ulimit -n` 调大、TCP 参数优化 |
| Tomcat 参数 | 调大 `maxConnections`、`acceptCount`、`maxThreads` |
| WebSocket Session | Session 数据精简，避免存储大对象 |
| 序列化 | Session 可通过 compression 压缩处理 |
| 内存 | 评估每连接占用，10w 连接预计占用 500MB-1GB |
| 异步 | 使用 Netty 事件驱动模型，减少线程切换 |
