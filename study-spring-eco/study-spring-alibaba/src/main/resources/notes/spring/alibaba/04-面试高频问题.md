## 问题一：Nacos vs Eureka 有什么区别？为什么选 Nacos？

**参考回答**：

Nacos 和 Eureka 都是服务注册发现组件，但 Nacos 的优势明显：

1. **CAP 模型**：Nacos 支持 AP + CP 双模式（临时实例走 AP/Distro 协议，永久实例走 CP/Raft 协议），Eureka 仅支持 AP。

2. **功能整合**：Nacos = 注册中心 + 配置中心，Eureka 只是注册中心，还需配合 Spring Cloud Config。

3. **通信协议**：Nacos 2.x 使用 gRPC 长连接（基于 HTTP/2），Eureka 使用 HTTP 短轮询，Nacos 性能和实时性高一个数量级。

4. **维护状态**：Eureka 2.x 已停止维护，Nacos 由阿里持续迭代。

5. **健康检查**：Nacos 支持 TCP/HTTP/MySQL 多种方式，Eureka 仅靠心跳续约。

6. **雪崩保护**：Nacos 有保护阈值（可配置），Eureka 有自我保护模式（默认开启）。

**选型结论**：国内新项目首选 Nacos，已用 Eureka 的旧项目可逐步迁移。

## 问题二：Sentinel 的滑动窗口是怎么实现的？

**参考回答**：

Sentinel 使用 **LeapArray**（环形数组）实现滑动窗口统计：

1. **数据结构**：固定长度的数组（默认 2 个），每个元素是一个 `MetricBucket`，包含 pass/block/complete/error/rt 等计数器。

2. **窗口划分**：整个数组覆盖 `sampleCount * intervalInMs` 的时间，例如 2 * 500ms = 1s。每个 Bucket 对应 `intervalInMs / sampleCount = 500ms`。

3. **Bucket 轮换**：每次访问时，根据当前时间计算所属的 Bucket：
   - 如果该 Bucket 不过期（当前时间在该 Bucket 窗口内），直接使用。
   - 如果过期，重置该 Bucket（利用 `AtomicLong` CAS 重置起始时间），然后将数据写入。

4. **QPS 统计**：遍历整个数组，累加所有未过期 Bucket 的 pass 计数。

**时间复杂度**：O(sampleCount)，典型值 O(2)，与请求量无关，性能极高。

**为什么不用滑动日志**：滑动日志需要维护请求时间戳列表（O(N) 空间和 O(logN) 查询），不适合高并发场景。

## 问题三：Seata AT 模式 vs TCC 模式，怎么选？

**参考回答**：

| 决策维度 | 选 AT | 选 TCC |
|---------|-------|--------|
| 业务复杂度 | CRUD 为主 | 涉及金额、库存、积分等高价值操作 |
| 改造成本 | 零侵入 | 需实现 Try/Confirm/Cancel 三接口 |
| 性能要求 | 一般 | 极高（无全局锁） |
| 数据一致性 | 最终一致（有全局锁） | 最终一致（无锁，但有悬挂风险） |
| 事务时长 | 短事务 (<5s) | 不限 |

**推荐策略**：
- 90% 场景用 AT 模式，零侵入、自动回滚。
- 涉及资金、扣库存等需要精确控制的场景用 TCC。
- TCC 需要注意**空回滚**（Try 未执行但收到 Cancel）和**悬挂**（Cancel 先于 Try 到达）。

## 问题四：Feign 的底层原理是什么？

**参考回答**：

Feign 的核心是 **JDK 动态代理** + **Contract 契约解析** + **Encoder/Decoder**：

1. **`@FeignClient` 注解解析**：Feign Builder 读取注解的 `name`/`url`/`fallback` 等属性。

2. **动态代理创建**：`ReflectiveFeign.newInstance()` 调用 `Proxy.newProxyInstance()`，创建接口的代理对象。所有方法调用被 `FeignInvocationHandler.invoke()` 拦截。

3. **方法调用处理**：
   - `MethodHandler`（通过 `SynchronousMethodHandler`）处理每次调用。
   - `Contract` 解析 `@RequestMapping`/`@GetMapping` 等注解，构建 `RequestTemplate`。
   - `Encoder` 将参数编码为 HTTP Body。
   - `RequestInterceptor` 链式添加 Header（如 Token 传递）。
   - `Client`（默认 `HttpURLConnection`，可替换为 OkHttp/Apache HttpClient）发送 HTTP 请求。

4. **负载均衡**：Spring Cloud LoadBalancer（替代已弃用的 Ribbon）在发送前选择合适的服务实例。

5. **熔断降级**：`FeignCircuitBreaker` 集成 Sentinel/Hystrix，调用失败时走 Fallback。

**一句话总结**：Feign = 接口注解 + JDK 动态代理 + HTTP 客户端 + 负载均衡。

## 问题五：Sentinel 热点参数限流是怎么实现的？

**参考回答**：

热点参数限流用于针对**特定参数值**进行精细化限流，例如对热门商品 ID `10001` 单独限流。

**实现原理**：

```
@SentinelResource(value = "getItem", blockHandler = "handleBlock")
public Item getItem(@RequestParam Long itemId) { ... }

// 规则配置
ParamFlowRule rule = new ParamFlowRule("getItem")
    .setParamIdx(0)              // 对第 0 个参数限流
    .setCount(10);               // 默认 QPS=10

// 对特定参数值设置例外
ParamFlowItem item = new ParamFlowItem()
    .setObject("10001")          // 热门商品 ID
    .setClassType(long.class.getName())
    .setCount(2);                // 热点商品 QPS=2
rule.setParamFlowItemList(Collections.singletonList(item));
```

**核心数据结构**：
- 为每个参数值维护一个独立的滑动窗口计数器。
- 先检查特定值是否有例外规则，有则按例外规则限流，否则按默认规则。
- 使用 `ConcurrentHashMap<Object, TokenBucket>` 存储每个参数值的统计。

**典型场景**：秒杀商品、热门接口参数、单个用户频率限制。

## 问题六：Nacos 配置优先级是如何确定的？

**参考回答**：

Spring Boot 的 `PropertySource` 有固定的加载顺序，越靠后的优先级越高（会覆盖前面的）：

```
1. 命令行参数           --server.port=9090              ← 最高优先
2. 环境变量             SERVER_PORT=9090
3. Nacos 远程配置        DataId=application-dev.yml      ← 覆盖本地配置
4. application.yml      classpath:/application.yml
5. bootstrap.yml        (Spring Cloud 2020 起已移除)      ← 最低优先
```

**Nacos 内部的配置优先级**（多个 DataId 时）：

```
1. ${spring.application.name}-${profile}.${extension}    (如: order-service-dev.yml)
2. ${spring.application.name}.${extension}               (如: order-service.yml)
3. ${spring.application.name}                            (如: order-service)
4. application-${profile}.${extension}                   (如: application-dev.yml)
5. application.${extension}                              (如: application.yml)
```

**Extension 优先级**：`yaml > yml > properties`

**共享配置 (shared-configs)** 和 **扩展配置 (extension-configs)** 的优先级低于主配置。

**重要**：`refresh-enabled: true` 时，Nacos 配置变更会触发 `RefreshScope` 刷新，但需要 `@RefreshScope` 或 `@ConfigurationProperties` 才能生效。
