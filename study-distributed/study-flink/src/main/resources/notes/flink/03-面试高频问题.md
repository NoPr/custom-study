## 问题 1：Watermark 如何解决乱序数据 + 延迟窗口触发条件？

```
Watermark = maxEventTimestamp - outOfOrdernessMs
```

**回答框架**：

1. **Watermark 定义**：告诉窗口"时间戳小于 Watermark 的数据都已到达"
2. **BoundedOutOfOrderness**：允许最大乱序延迟（如 3s）
3. **窗口触发**：`watermark >= window.end` 时触发计算
4. **延迟数据处理**：`allowedLateness` + `SideOutput` 侧输出

**关键得分点**：
- Flink 用 Watermark 替代 Spark 的固定批次窗口
- EventTime 语义保证结果确定性，不受网络延迟影响
- Flink 1.12+ 支持 `WatermarkStrategy` + `withIdleness`

## 问题 2：Checkpoint vs Savepoint 区别及使用场景？

**核心区别表**：

| 维度 | Checkpoint | Savepoint |
|------|-----------|-----------|
| 触发方式 | 自动(JM 定时) | 手动(CLI/REST) |
| 目的 | 故障恢复 | 版本升级/代码迁移 |
| 生命周期 | 完成后自动清理 | 永久保留 |
| 状态格式 | 后端原生格式 | 标准化格式(可迁移) |
| 并行度 | 必须匹配 | 可修改 |
| UID 要求 | 可选 | 必须设置 |

**面试加分回答**：
- Checkpoint 底层是 Chandy-Lamport 异步分布式快照算法
- Barrier 对齐是 Exactly-Once 的核心
- RocksDB 支持增量 Checkpoint（只上传新增 SST 文件）
- Savepoint 触发前必须为所有 Operator 设置 `uid()`

## 问题 3：Flink 怎么保证 Exactly-Once 语义？

**两层回答**：

**A. Flink 内部 Exactly-Once（状态一致性）**：
1. Checkpoint 机制：JM 定期注入 Barrier
2. Barrier 对齐：多输入 Operator 等待所有上游 Barrier 到达
3. 故障恢复：从最近 Checkpoint 恢复状态 + Source 重置 Offset

**B. 端到端 Exactly-Once（TwoPhaseCommitSink）**：
1. Phase 1 (pre-commit)：CK 完成时 Sink 预提交
2. Phase 2 (commit)：JM 确认 CK 后 Sink 正式提交
3. 适用 Sink：Kafka 0.11+、FileSystem

**关键条件**：
- `env.enableCheckpointing(5000, EXACTLY_ONCE)`
- Source 支持重置（Kafka）、Sink 支持 2PC

## 问题 4：Flink 反压 (backpressure) 机制是怎样的？

**本质**：下游处理慢 --> TCP 缓冲区满 --> Netty 阻塞 --> 逐级反压到 Source

**Flink 1.5+ Credit-Based Flow Control**：
- 下游明确通告可用 Buffer 数量（credit）
- 上游仅在有 credit 时发送数据
- 优势：无 Head-of-Line Blocking，上游立即可感知下游瓶颈

**反压定位三步法**：
1. WebUI 看每个 Operator 的 BackPressure 状态（HIGH/LOW/OK）
2. 从 Sink 往上追踪，第一个 HIGH 就是瓶颈
3. 解决：增加并行度、减少 KeyBy 倾斜、Sink 批量写入

## 问题 5：Flink vs Spark Streaming vs MR 对比

| 维度 | Flink | Spark Streaming | MapReduce |
|------|-------|----------------|-----------|
| 计算模型 | 流式计算（实时） | 微批处理（MicroBatch） | 批处理（Batch） |
| 延迟 | 毫秒级 (< 100ms) | 秒级（亚秒~几秒） | 分钟~小时 |
| 数据抽象 | Stream（无界） | DStream（微批 RDD） | HDFS 文件（有界） |
| 状态管理 | KeyedState/OperatorState | updateStateByKey | 无状态 |
| Checkpoint | Chandy-Lamport 异步 | WAL/Lineage 重算 | 无 |
| Exactly-Once | 支持（2PC Sink） | 需外部幂等 | 不支持 |

**一句话总结**：
- **Flink**：真正的流处理，来一条处理一条，毫秒级延迟
- **Spark Streaming**：微批处理，N 秒一批，不是真正的实时
- **MapReduce**：批处理，分 Map 和 Reduce 两阶段，离线计算

## 额外高频问题速查

### Q6：Flink 支持哪些时间语义？
EventTime / ProcessingTime / IngestionTime（不推荐，已被废弃）

### Q7：Flink 如何处理迟到数据？
`allowedLateness` 延长窗口等待时间 + `SideOutput` 侧输出流收集

### Q8：Flink 的并行度怎么设置？
Source 并行 = Kafka 分区数，下游逐算子调优，`slotSharingGroup` 共享 Slot

### Q9：状态后端怎么选？
- 开发测试：HashMapStateBackend（内存，快）
- 生产海量：EmbeddedRocksDBStateBackend（磁盘，TB 级，增量 CK）

### Q10：Slot 与 Task 的关系？
每个 Slot 运行一个完整的 Pipeline，同一个 Job 的不同 Task 可以共享 Slot
